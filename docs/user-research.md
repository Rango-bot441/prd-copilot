# 用户研究：PRD Copilot

## 1. 研究目标

本项目要解决的问题是：当用户只有一个模糊产品想法时，如何帮助他快速补齐 PRD 前置思考，并生成一份可进入评审的结构化草稿。

## 2. 目标用户画像

| 用户 | 背景 | 痛点 | 诉求 |
| --- | --- | --- | --- |
| 产品求职者 | 需要向 HR 和面试官证明 PRD 能力 | 只有 Demo 不足以证明产品基本功 | 需要展示需求分析、PRD、指标和迭代思路 |
| 初级产品经理 | 经常接到模糊业务需求 | 不知道该追问什么 | 需要标准化澄清框架 |
| 产品运营 | 更熟悉业务，但不熟悉 PRD 写法 | 想法难以转成研发可理解的文档 | 需要把业务语言转成产品语言 |
| 独立开发者 | 想快速做 MVP | 容易边做边改，范围失控 | 需要先明确 MVP 和验收标准 |

## 3. 核心痛点

- 需求只有一句话，缺少用户、问题、场景和指标。
- PRD 容易写成模板填空，而不是问题澄清后的方案。
- 评审时才发现边界、异常状态、验收标准缺失。
- 产品求职作品集经常只有页面 Demo，缺少产品思考过程。

## 4. 用户任务

| 任务 | 重要性 | 产品响应 |
| --- | --- | --- |
| 判断需求是否成熟 | 高 | 成熟度评分 |
| 找出缺失信息 | 高 | 缺失要素列表 |
| 知道下一步问什么 | 高 | 建议追问 |
| 形成需求卡 | 高 | 用户、问题、目标、MVP、指标 |
| 生成 PRD 草稿 | 高 | PRD Builder |
| 拿去评审或展示 | 中 | 复制和导出 Markdown |

## 5. 用户旅程

```text
产生想法
  → 不确定是否能写 PRD
  → 输入一句话需求
  → 查看需求成熟度
  → 发现缺失信息
  → 根据追问补充背景
  → 生成 PRD 草稿
  → 复制或导出文档
  → 进入评审 / 放入作品集
```

## 6. 产品洞察

### 洞察 1：PRD 能力的核心是澄清，而不是排版

面试官真正关注的是候选人能否识别模糊需求中的缺口，而不是文档格式是否好看。

### 洞察 2：好的 PRD 应该自带验证方式

如果需求没有指标和验收标准，就很难判断功能是否有效，也难以推动跨团队协作。

### 洞察 3：作品集需要展示过程

对求职者来说，Demo 只是结果。用户研究、需求卡、PRD、指标和迭代计划才能展示产品思考。

## 7. 设计影响

| 研究结论 | 设计响应 |
| --- | --- |
| 用户不知道缺什么 | 展示缺失 PRD 要素 |
| 用户不知道问什么 | 自动生成追问问题 |
| 用户需要快速展示能力 | 三栏工作台同时展示输入、澄清、PRD |
| 用户需要作品集材料 | README 和 docs 文档完整沉淀 |
